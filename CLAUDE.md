```bash

npm run build        # Production build to ./dist/
npm run generate     # Parse .meta files and regenerate all search JSON indexes
npm run update       # generate + build (full rebuild)
npm run deploy       # build + deploy to Cloudflare Workers via Wrangler
npm run preview      # build + local Cloudflare Workers preview via Wrangler
npm run notion-pull  # Fetch Done pages from Notion REST API → src/data/notion/

npm run publish-content # notion-pull --force + notion-sync --force + update
```

## Architecture

This is an **Astro 5** static knowledge base site deployed as a **Cloudflare Worker** (via `@astrojs/cloudflare` adapter). It uses **Tailwind CSS v4** (via `@tailwindcss/vite` plugin, not the legacy Astro integration).

### Content Structure

New content is added exclusively via `/notion-sync`. Pages live at:

```
src/pages/[category]/[section]/[slug].astro   ← generated by /notion-sync
src/pages/[category]/[section]/[slug].meta    ← generated by /notion-sync
```

Some legacy markdown content exists at `src/content/knowledge/[category]/[section]/[slug].md` (served by the catch-all route), but this is **not** the authoring path for new content.

Categories and sections (defined in `src/data/site-config.json`):
```
ciberseguridad/   (fundamentos/, criptografia/, redes-seguras/)
redes/            (fundamentos/, protocolos/, seguridad-redes/)
sistemas/         (operativos/, virtualizacion/, gobernanza/, monitorizacion/)
programacion/     (fundamentos/)
cloud/            (aws/)
cert/             (certification study pages — legacy path only)
```

### Site Config (source of truth)

`src/data/site-config.json` is the single source of truth for categories, sections, colors, and Notion DB IDs. Key uses:
- `generateSimple.cjs` reads categories from here (no hardcoded array)
- Copied to `public/site-config.json` at build time (for category pages to fetch at runtime)
- `Navbar.astro` imports it statically to generate nav links
- Top-level category pages are resolved by `src/pages/[category].astro` from `site-config.json`

To **add a section**: edit `site-config.json` → `npm run generate`. No code changes needed.
To **add a top-level category**: add it in `site-config.json` (or let `notion-pull` auto-create it). No manual `src/pages/[cat].astro` file is needed.

### Metadata & Search System

`src/scripts/generateSimple.cjs` is the core build script that:
1. Scans all `.meta` files AND `src/content/knowledge/**/*.md` frontmatter
2. Writes per-topic JSON to `src/data/categories/[cat]/[section]/[slug].json`
3. Generates search indexes to `public/`:
   - `search-index.json` — global index of all pages
   - `search-[category].json` — per-category index
   - `search-cert-[CODE].json` — per-certification index (e.g. `search-cert-SY0-701.json`)

**Always run `npm run generate` after adding/editing `.meta` files.**

`.meta` file format:
```
title=Topic Title
slug=url-slug
icon=🔧
status=completed
order=1
description=Short description
tags=tag1,tag2,tag3
certifications=SY0-701:4.5:General Security Concepts,NSE4_FGT-7.2:4.1:Security Fabric
```

Certification config lives in `src/data/certifications/index.json`.

### Layout & Components

- `src/layouts/WikiLayout.astro` — base layout for all content pages; wraps in `<html>`, includes Navbar, header with search, footer, dark mode toggle
- `src/components/Navbar.astro` — top navigation between categories
- `src/components/Quiz.astro` — reusable quiz component; consumes a JSON file from `src/data/quiz/[topic].json`
- Category index pages are generated by `src/pages/[category].astro` from `site-config.json`

### Adding a New Content Page

**The only method is `notion-pull` + `notion-sync` (CLI).**

Workflow:
1. Author content in Notion (Status: Draft)
2. Mark as `Done` in Notion
3. `npm run notion-pull` — fetches all Done pages from Notion REST API (requires `NOTION_API_KEY` in `.env`), saves to `src/data/notion/[cat]/[sec]/[slug].json`
4. Run `npm run notion-sync` — reads local JSONs in `src/data/notion/`, generates `.astro` + `.meta` for any new pages
5. `npm run dev` — review locally
6. `npm run deploy` — publish to Cloudflare

**Usage:**
```
npm run notion-pull                    # pull from Notion (global DB mode)
npm run notion-pull -- --force         # overwrite existing local JSONs
npm run notion-pull -- --no-cleanup    # pull without deleting local files from pages no longer Done
npm run notion-sync                    # generate pages from local JSONs
npm run notion-sync -- ciberseguridad  # generate one category only
npm run notion-sync -- --force         # overwrite existing .astro pages
npm run publish-content                # one command: pull + sync + generate + build
```

**Notion database (DB mode):**

`src/data/site-config.json` has a single global `"notionDbId"` (currently `"31bec7e74fc08059b87ec747f92bee36"`) and `notion-pull` uses only that value.

Per-category `notionDbId` entries were removed from categories in `site-config.json` and are no longer part of the pull logic.

Each row in the DB uses:
| Notion field | Maps to |
|---|---|
| `Categoría` (select) | `category` (kebab-case normalized) |
| `Subcategoría` (select) | `section` (kebab-case normalized) |
| `Status` (status) | only `Done` is fetched |
| `slug`, `icon`, `description`, `tags`, `certifications`, `order` | standard metadata |

`notion-pull` auto-adds missing categories/sections to `site-config.json` (with default gray color + 📁/📄 icons).
`notion-pull` also runs cleanup by default: if a page is no longer `Done`, related files are removed from `src/data/notion/` and generated `src/pages/...` outputs. Use `--no-cleanup` to skip this behavior.

If `npm run notion-pull` returns `object_not_found`, run `npm run notion-pull -- --list` and verify the DB/data source is shared with the integration. Notion URLs can expose a different ID than the queryable data source ID in API v2025+.

Only pages with `Status = Done` are processed. Pages that already exist in `src/pages/` **or** `src/content/knowledge/` are skipped.

**Note:** `notion-pull` solves the MCP Notion limitation — the MCP `notion-fetch` tool can only do semantic search over page bodies, not query database rows by property. The REST API (`@notionhq/client`) filters by `Status = Done` directly.

### Quiz Component Usage

```astro
---
import WikiLayout from '../../layouts/WikiLayout.astro';
import Quiz from '../../components/Quiz.astro';
import quizData from '../../data/quiz/[topic].json';
---
<WikiLayout title="..." description="...">
  <!-- page content -->
  <Quiz questions={quizData.questions} title={quizData.title} />
</WikiLayout>
```

Quiz JSON schema: `{ title: string, questions: [{ question, options: string[4], correct: 0-3, explanation }] }`

### Deployment

The site deploys to `https://kb.villaarreola.com` via Cloudflare Workers. Config is in `wrangler.jsonc`. The Astro cloudflare adapter outputs `dist/_worker.js/index.js` as the worker entry point with static assets bound as `ASSETS`.
